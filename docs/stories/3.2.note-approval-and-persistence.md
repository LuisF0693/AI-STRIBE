## Story
**As a** médico,
**I want** aprovar a nota SOAP revisada com um toque e ter a versão final salva com segurança,
**so that** a nota clínica seja preservada com rastreabilidade completa (quem aprovou, quando) e fique disponível para exportação PDF (Story 3.3).

## Acceptance Criteria
1. Endpoint `PATCH /api/v1/notas/:id` salva edições de rascunho do médico — não altera `status`, apenas persiste o `soap_json` editado e os `cids_sugeridos` selecionados
2. Endpoint `POST /api/v1/notas/:id/aprovar` finaliza a nota: atualiza `status` de `'draft'` para `'approved'`, registra `approved_by` (medico_id) e `approved_at` (timestamp UTC)
3. A aprovação cria automaticamente um snapshot imutável em `notas_versoes` com `soap_json` final e `editado_por`
4. Idempotência: chamar `POST /api/v1/notas/:id/aprovar` em nota já aprovada retorna 200 com o estado atual — sem erro, sem duplicação
5. RLS Supabase: médico só pode editar ou aprovar notas associadas às suas próprias consultas — `supabaseAdmin` nunca usado no client mobile
6. Transição de status respeita a state machine (`updateNotaStatus()`) — direto UPDATE no banco sem validação é bloqueado pelo padrão da codebase
7. Após aprovação bem-sucedida, Supabase Realtime notifica o mobile com evento `nota:approved` (canal `consulta:{id}`)
8. Mobile: ao receber confirmação de aprovação, Zustand store é atualizado (`nota.status = 'approved'`) e usuário é redirecionado para a tela de exportação PDF
9. Histórico de versões: todas as edições intermediárias (PATCH) são registradas em `notas_versoes` para auditoria LGPD
10. Retorno de erro padronizado (`ApiError`) para: nota não encontrada (404), nota já aprovada sem mudanças (200 idempotente), acesso negado (403)

## Escopo

**IN (esta story):**
- Endpoint PATCH para salvar rascunho editado
- Endpoint POST para aprovação final da nota
- State machine `updateNotaStatus()` com validação de transição
- Histórico de versões em `notas_versoes`
- Notificação Realtime `nota:approved`
- Migration 007 (approved_by, approved_at, pdf_status)

**OUT (fora desta story):**
- Geração de PDF → Story 3.3
- Interface de revisão da nota → Story 3.1
- Reprovação/rejeição de nota → não previsto no MVP
- Aprovação por terceiros (auditoria multi-médico) → pós-MVP
- Integração com PEPs → Epic 5

**Estimativa de Complexidade:** `M` (Medium) — backend puro, bem delimitado; RLS e idempotência são os pontos críticos

## Dev Notes

### Schema Existente (Story 2.3)
```sql
-- Tabela notas (já existe):
CREATE TABLE notas (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  consulta_id UUID REFERENCES consultas(id),
  transcricao_id UUID REFERENCES transcricoes(id),
  soap_json JSONB NOT NULL,
  cids_sugeridos JSONB,
  status TEXT CHECK (status IN ('draft', 'reviewed', 'approved')),
  baixa_confianca BOOLEAN DEFAULT false,
  versao INTEGER DEFAULT 1,
  created_at TIMESTAMPTZ DEFAULT NOW()
);
-- notas_versoes (já existe em 2.3):
CREATE TABLE notas_versoes (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  nota_id UUID REFERENCES notas(id),
  soap_json JSONB NOT NULL,
  versao INTEGER,
  editado_por UUID REFERENCES medicos(id),
  created_at TIMESTAMPTZ DEFAULT NOW()
);
```

### Migration 007 (adições desta story)
```sql
-- Adicionar na tabela notas:
ALTER TABLE notas
  ADD COLUMN approved_by UUID REFERENCES medicos(id),
  ADD COLUMN approved_at TIMESTAMPTZ,
  ADD COLUMN pdf_status TEXT CHECK (pdf_status IN ('none', 'pending', 'processing', 'ready', 'failed')) DEFAULT 'none';

-- RLS policy atualizada:
CREATE POLICY "Médico acessa suas notas" ON notas
  FOR ALL USING (
    consulta_id IN (
      SELECT id FROM consultas WHERE medico_id = auth.uid()
    )
  );
```
