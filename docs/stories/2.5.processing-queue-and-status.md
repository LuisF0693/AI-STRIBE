## Story
**As a** produto,
**I want** uma fila de processamento robusta e tolerante a falhas para gerenciar transcrição e geração de notas,
**so that** nenhuma consulta seja perdida, o médico sempre receba feedback de status e o sistema escale sem degradação.

## Acceptance Criteria
1. Fila BullMQ com 3 filas separadas: `audio-upload`, `transcription`, `note-generation` — cada uma com concorrência configurável
2. Jobs expiram após 24h se não processados (TTL) — médico notificado por push notification
3. Dashboard de status acessível via API REST: `GET /api/consultas/:id/status` retorna estado completo do pipeline
4. Dead Letter Queue (DLQ) para jobs que falharam após todas as tentativas — alertas automáticos para o time
5. Métricas de fila expostas: jobs_waiting, jobs_active, jobs_completed, jobs_failed — compatível com Prometheus
6. Retry automático com backoff exponencial configurável por fila (audio: 3x, transcription: 3x, note: 2x)
7. Idempotência garantida — reprocessar o mesmo `consulta_id` não cria duplicatas em nenhuma tabela
8. Health check endpoint `GET /health/queues` retorna status de cada fila (usado pelo CI/CD e monitoramento)
9. Graceful shutdown: ao receber SIGTERM, workers completam jobs em andamento antes de encerrar (max 30s)
10. Logs estruturados (JSON) para cada transição de estado de job — compatível com Datadog/CloudWatch

## Dev Notes

### Contexto Técnico
- Redis: Upstash Redis (serverless, região sa-east-1 disponível, plano free: 10K comandos/dia)
- BullMQ requer Redis 6.2+ — Upstash suporta
- Para prod com volume alto: migrar para Redis dedicado (Railway, Render, ou EC2)

### Configuração de Filas
```typescript
// packages/api/src/queues/config.ts
export const QUEUE_CONFIG = {
  audioUpload: { concurrency: 10, attempts: 3, backoff: { type: 'exponential', delay: 5000 } },
  transcription: { concurrency: 5, attempts: 3, backoff: { type: 'exponential', delay: 10000 } },
  noteGeneration: { concurrency: 10, attempts: 2, backoff: { type: 'exponential', delay: 5000 } },
}
```

### Flow de Status por Consulta
```
consulta criada → upload: pending
upload concluído → upload: completed | transcription: pending
transcrição iniciada → transcription: processing
transcrição concluída → transcription: completed | note: pending
nota iniciada → note: processing
nota pronta → note: completed → notificar médico
qualquer falha → {stage}: failed → DLQ → alerta
```

### Dependências de Infra
- Story 1.1: CI/CD deve incluir health check `GET /health/queues` antes de deploy
- Redis URL configurada como secret no GitHub Actions e variável de ambiente no Vercel/Railway

### Testing
- Localização: `packages/api/src/queues/__tests__/`
- Framework: Jest + Testcontainers (Redis container) para integração real
- Cenários obrigatórios: job duplicado, falha de rede durante upload, SIGTERM durante processamento
